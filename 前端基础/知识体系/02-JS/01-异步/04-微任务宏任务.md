# 微任务 / 宏任务

## 微任务 MicroTask

微任务几乎全部由 JavaScript 语言标准 提供：

- `Promise.then / .catch / .finally `回调

- async/await 的后续代码（本质上是 Promise.then）
- queueMicrotask(fn)（ES2019 引入，最纯的微任务）
- MutationObserver 回调（DOM 变化观察器）
- process.nextTick（`Node.js` 特有，比微任务还早）

一句话：语言标准（ECMAScript）提供的异步回调大多是微任务。





## 宏任务 macroTask

浏览器/Node 环境中常见的宏任务来源（按优先级大致排序）：

- script 标签整体代码（主模块）
- setTimeout / setInterval 的回调
- setImmediate（`Node.js` 特有）
- I/O 操作回调（网络请求完成、文件读写完成）
- UI 渲染（浏览器特有，在宏任务之间插入）
- requestAnimationFrame 回调（浏览器特有，渲染相关）
- postMessage / MessageChannel（常用于模拟微任务，但本身是宏任务）
- DOM 事件回调（click、scroll、keydown 等）
- Ajax

一句话：宿主环境（浏览器/Node）提供的异步 API 大多是宏任务。


## event loop 和 DOM 渲染

Event Loop 和 DOM 渲染 是前端性能优化、理解“为什么 setTimeout 回调比 Promise.then 慢”
“为什么修改 data 后立即读取 DOM 还是旧值”等现象的核心知识点。


### 浏览器渲染流程

现代浏览器（Chrome、Safari、Firefox 等）以 每秒 60 帧（约 16.67ms 一帧）为目标渲染页面，每一帧大致流程如下：

```text

1. JS 执行（执行同步代码 + 宏任务回调）
   ↓
2. 计算样式（Recalculate Style）
   ↓
3. 布局（Layout / Reflow）
   ↓
4. 绘制（Paint）
   ↓
5. 合成（Composite） → 最终显示到屏幕

```

关键点：浏览器会在合适的时机插入“渲染步骤”，但不是每执行一行 JS 就渲染一次（那样太浪费性能）。

### Event Loop 在哪里插入渲染

标准的事件循环伪代码（HTML 规范 + 浏览器实现）：


```text

while (true) {  // 事件循环主循环

  // A. 执行当前宏任务（或主线程脚本）
  执行一个宏任务（script / setTimeout 回调 / 用户点击事件 / I/O 回调 等）

  // B. 清空微任务队列（清到空为止）
  while (微任务队列不为空) {
    执行微任务（Promise.then / await 后续 / queueMicrotask / MutationObserver）
  }

  // C. 执行渲染更新（如果需要且时机合适）
  如果需要渲染（有样式/布局/绘制变更） 且 帧时间还没用完
    → 执行渲染步骤（样式 → 布局 → 绘制 → 合成）

  // D. 下一帧
}



```

浏览器会在以下几个时机尝试渲染（不保证每轮都渲染）：

- 微任务队列清空后（最常见、最可靠的渲染点）
- 当前宏任务执行完，且没有更多微任务
- requestAnimationFrame 回调执行前/后
- 浏览器认为时机合适（通常 16ms 帧间隔内）

Event Loop 决定 JS 代码执行顺序，浏览器渲染决定 DOM 何时更新。


```html


<!DOCTYPE html>
<html>
<body>
  <div id="box" style="width: 100px; height: 100px; background: red;">
  </div>

  <script>
    const box = document.getElementById('box');

    // 1. 同步代码：修改 DOM 样式
    console.log('1. 同步代码开始');
    box.style.background = 'blue'; // 修改 DOM，但此时页面还没渲染

    // 2. 微任务：Promise.then
    Promise.resolve().then(() => {
      console.log('2. 微任务执行');
      // 能拿到最新的 'blue'（DOM 已更新，但页面未渲染）
      console.log('微任务中读取背景色：', box.style.background); 
    });

    // 3. requestAnimationFrame：渲染前执行
    requestAnimationFrame(() => {
      console.log('3. requestAnimationFrame 执行（渲染前）');
      box.style.background = 'green'; // 再次修改，会在本次渲染中生效
    });

    // 4. 宏任务：setTimeout
    setTimeout(() => {
      console.log('4. 宏任务执行（setTimeout）');
      // 此时页面已渲染，是 'green'
      console.log('宏任务中读取背景色：', box.style.background); 
    }, 0);

    console.log('5. 同步代码结束');
  </script>
</body>
</html>

```


## 微任务为什么执行更早


**微任务队列是在当前“执行上下文”（execution context）结束时、渲染之前必须被清空的，而宏任务队列是下一轮循环才开始处理。**


1. 事件循环最简洁的伪代码（现代浏览器规范版）

```text

while (true) {
  // ──────────────── 当前这一轮循环开始 ────────────────

  // A. 从宏任务队列取出一个任务（如果有的话）
  let task = macrotaskQueue.pop()   // 可能是 setTimeout 回调、click 事件、I/O 回调等
  if (task) {
    执行 task 的回调函数           // 这里开始一个新的执行上下文
  }

  // B. 执行完当前宏任务后，**立即**清空微任务队列（清到空为止！）
  while (microtaskQueue 不为空) {
    let microtask = microtaskQueue.shift()
    执行 microtask 的回调           // Promise.then、await 后续、queueMicrotask 等
  }

  // C. （浏览器）尝试渲染（如果有需要更新的内容且时机合适）
  performRenderingIfNeeded()        // 样式重计算 → 布局 → 绘制 → 合成

  // ──────────────── 当前这一轮循环结束 ────────────────
}

// 微任务队列的清空是嵌套在当前宏任务内部的
// 只要当前宏任务的回调函数执行完毕，浏览器就会立刻、反复执行微任务，直到微任务队列为空
// 只有微任务队列彻底为空，浏览器才会考虑去执行下一个宏任务，或者进行渲染
```





因为在事件循环的每一轮中，微任务队列被设计为“当前执行上下文必须清空的收尾工作”，
而宏任务队列是“下一轮全新的执行上下文的开始”。

- 清空微任务是当前轮的强制要求
- 执行下一个宏任务是下一轮的开始


## 示例


宏任务： DOM 渲染后触发，如 setTimeout

微任务： DOM 渲染前触发，如 Promise


```js

const $p1 = $('<p>一段文字</p>')
const $p2 = $('<p>一段文字</p>')
const $p3 = $('<p>一段文字</p>')


$('#container')
    .append($p1)
    .append($p2)
    .append($p3)

// 微任务： DOM 渲染前触发
Promise.resolve().then(() => {
  console.log('length1',
    $('#container').children().length)
  alert('promise then') // DOM 渲染了吗
})


// 宏任务：DOM 渲染后触发
setTimeout(() => {
  console.log('length2',
    $('#container').children().length)
  alert('setTimeout') // DOM 渲染了吗
})
```
